To use `dash`, follow these steps after you've [installed it][01]:

1. Run

   ```bash
   gl-dash
   ```

2. Press <kbd>?</kbd> for help.
3. Run `gl-dash --help` for more info:

   ```bash
   Usage:
     gl-dash [flags]

   Flags:
     -c, --config string   use this configuration file (default lookup: a .gl-dash.yml file if inside a git repo, $GL_DASH_CONFIG env var, or if not set, $XDG_CONFIG_HOME/gl-dash/config.yml)
         --debug           passing this flag will allow writing debug output to debug.log
     -h, --help            help for Dash
   ```

## Flags

### `--config`

Specify the path to a configuration file to use for the dashboard. If the configuration file
doesn't exist or is invalid, `dash` returns an error.

```bash
gl-dash --config path/to/configuration/file.yml
```

| Aliases |  Type  | Default                                                                                                                   |
| :------ | :----: | :------------------------------------------------------------------------------------------------------------------------ |
| `-c`    | String | `.gl-dash.yml` file if inside a git repo, `$GL_DASH_CONFIG` env var, or if not set, `$XDG_CONFIG_HOME/gl-dash/config.yml` |

If you don't specify this flag, `dash` uses the default configuration.
If the file doesn't exist, `dash` will create it.

The location of the default configuration file depends on your system:

1. If `$GL_DASH_CONFIG` is a non-empty string, `dash` will use this file for
   its configuration.
2. If `$GL_DASH_CONFIG` isn't set and you're in a git repository, it will look for `.gl-dash.yml` or `.gl-dash.yaml`
   in the repository root.
3. If neither of the above applies, then:
   - If `$XDG_CONFIG_HOME` is a non-empty string, the default path is `$XDG_CONFIG_HOME/gl-dash/config.yml`.
   - If `$XDG_CONFIG_HOME` isn't set, then:
     - On Linux and macOS systems, the default path is `$HOME/.config/gl-dash/config.yml`.
     - On Windows systems, the default path is `%USERPROFILE%\.config\gl-dash\config.yml`.

For more information about authoring configurations, see [Configuration][02].

### `--debug`

Specify whether `dash` should write logs to the `debug.log` file in the current directory. By
default, `dash` doesn't output debug information.

```bash
gl-dash --debug
```

| Aliases |  Type   | Default |
| :------ | :-----: | :------ |
| (None)  | Boolean | `false` |

When you use this flag, `dash` creates the `debug.log` file in the current directory if it doesn't exist. If the file does exist, `dash` appends new log entries to it.

### `--help`

Use this flag to display the help information for `dash` in the terminal. If you specify this
flag, `dash` ignores all other flags. It only displays the help information.

```bash
gl-dash --help
```

| Aliases |  Type   | Default |
| :------ | :-----: | :------ |
| `-h`    | Boolean | `false` |

### `--version`

Use this flag to display the version information for `dash` in the terminal. If you specify this
flag with the `--config` or `--debug` flags, `dash` ignores them. It only displays the version
information.

```bash
gl-dash --version
```

| Aliases |  Type   | Default |
| :------ | :-----: | :------ |
| `-v`    | Boolean | `false` |

When you use this flag, `dash` emits the following information:

```text
gl-dash version <version>
commit: <commit_sha>
built at: <build_timestamp>
built by: <build_user>
goos: <operating_system>
goarch: <cpu_architecture>
```

- `<version>` is the binary's semantic version without a `v` prefix.
- `<commit_sha>` is the exact commit SHA the binary was built from.
- `<build_timestamp>` is the UTC date and time when the binary was built.
- `<build_user>` is who built the binary. For official releases, this is always `goreleaser`.

For example, the version information for a release on Windows with an x64 processor
is:

```text
gl-dash version 3.7.7
commit: 6ce3f89ab0d73dd88e359133699d1cf920f88699
built at: 2023-04-15T08:42:16Z
built by: goreleaser
goos: windows
goarch: amd64
```

## Default Keybindings

When you use `dash`, it displays the dashboard as a terminal UI (TUI). In the TUI, you can use
several commands by pressing key combinations to navigate and interact with the dashboard.

You can press the <kbd>?</kbd> key to toggle the help menu in the UI. The help menu lists the available
keybindings for the current context.

You can press the <kbd>q</kbd> key to quit the dashboard and return to your normal terminal view.

For more information about the keybindings for the dashboard, see [Keybindings][04].
